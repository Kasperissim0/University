Growth of Functions Table

Exercise 1-1 (partial) — CLRS Ch. 1  ·  Lecture slides pp. 54–55

What this asks. The lecture (slide p. 54) presents a "Function Growth Cheat Sheet" ordering growth rates from slowest to fastest: $1 \prec \lg n \prec \sqrt{n} \prec n \prec n\lg n \prec n^2 \prec \cdots \prec 2^n \prec n!$, where $\prec$ means "grows strictly slower than," i.e., is little-$o$ of the right side (slide p. 52).

This table makes that ordering vivid and practical: assuming each elementary operation takes 1 microsecond — consistent with the lecture's computational model (slides pp. 2–6) where each statement takes a constant time — what is the largest $n$ solvable within time $T$? We solve $f(n) = T$ for $n$.

Why this matters. Slide p. 55 ("Tips and Examples") states: "Focus on the fastest growing term of a function ignoring the constant coefficient." The table shows why: the difference between $n^2$ and $2^n$ is not a constant factor — it is the difference between solving 1,000 vs. 19 inputs per second.

Lecture refs: Function Growth Cheat Sheet (p. 54); tips and examples (p. 55); computational model (pp. 2–6); Big-$O/\Omega/\Theta$ definitions used to classify rows (pp. 50–52).
Derivation

Let $T$ = time budget in microseconds. The largest solvable $n$ satisfies $f(n) \leq T$:

$$\lg n \leq T \;\Rightarrow\; n = 2^T \qquad\qquad n \leq T \;\Rightarrow\; n = T$$ $$n\lg n \leq T \;\Rightarrow\; n \text{ found by binary search} \qquad\qquad n^2 \leq T \;\Rightarrow\; n = \lfloor\sqrt{T}\rfloor \qquad\qquad 2^n \leq T \;\Rightarrow\; n = \lfloor\log_2 T\rfloor$$

Time limits: $10^6\,\mu s$ (1 s), $6{\times}10^7$ (1 min), $3.6{\times}10^9$ (1 hr), $8.64{\times}10^{10}$ (1 day), ${\approx}\,3.16{\times}10^{15}$ (1 century).

Fibonacci Recursion & Induction

Strong induction · Call tree · Runtime lower bound  ·  Lecture slides pp. 26–36

Induction in algorithms. Slide p. 34 motivates induction directly: "Many algorithms are step-by-step — loops, recursion, array prefixes, growing structures. Typical claim: after step $i$, some property holds — which is exactly the structure of induction." For recursive algorithms, slide p. 17 states correctness is proved by induction over input size: $\forall n \in \mathbb{N},\, \forall$ valid inputs $x$ of size $n$: output of procedure(x) is correct.

Strong induction template (slide p. 31): Let $P$ be a predicate. Prove $P(0)$. Then suppose $n$ is arbitrary but fixed, assume $P(0), P(1), \ldots, P(n-1)$ are all true (strong induction hypothesis), and prove $P(n)$. The lecture notes the proof must not depend on the actual value of $n$ because $n$ is arbitrary.

Why strong induction here? The Fibonacci recurrence $F_n = F_{n-1} + F_{n-2}$ references the two preceding values, so we need both $P(n-1)$ and $P(n-2)$ in the inductive step. Ordinary induction only gives $P(n-1)$. Strong induction (slide p. 31) gives us all predecessors simultaneously.

Key hint. $\varphi = (1+\sqrt{5})/2$ satisfies $\varphi^2 = \varphi + 1$. Verify: $\varphi^2 = (6+2\sqrt5)/4 = (3+\sqrt5)/2 = 1 + (1+\sqrt5)/2 = 1 + \varphi$. ✓

Lecture refs: induction motivation (p. 34); correctness format (p. 17); ordinary induction template (p. 26); strong induction template (p. 31); finite domain variant (p. 32); induction example (pp. 28–30); bogus Fibonacci proof as homework warning (p. 33).
Part 1 — Proof by induction that fib(n) $= F_n$
int fib(int n) {
    if (n <= 1) return n;
    return fib(n-1) + fib(n-2);
}

Claim: For all $n \geq 0$, fib(n) returns $F_n$. p. 17 — correctness format

Proof — strong induction on $n$ p. 31 — strong induction template
Base Cases — $n = 0$ and $n = 1$

$n = 0$: Condition n <= 1 is true; returns $0 = F_0$. ✓

$n = 1$: Condition n <= 1 is true; returns $1 = F_1$. ✓

(Two base cases are needed because the inductive step uses both $P(n-1)$ and $P(n-2)$.)

Inductive Hypothesis

Suppose $n \geq 2$ is arbitrary but fixed. Assume that for every $k$ with $0 \leq k < n$, fib(k) returns $F_k$. (Note: the proof does not depend on the specific value of $n$, since $n$ is arbitrary — as required by slide p. 26.)

Inductive Step — prove for $n \geq 2$

Since $n \geq 2$, condition n <= 1 is false. The function executes return fib(n-1) + fib(n-2).

Since $n-1 < n$ and $n-2 < n$, the inductive hypothesis gives: fib(n-1) $= F_{n-1}$ and fib(n-2) $= F_{n-2}$.

$$\texttt{fib}(n) = F_{n-1} + F_{n-2} = F_n \qquad \text{(by definition of Fibonacci)} \quad \checkmark$$
□
Part 2 — Call tree for fib(5)

Each node labelled $k$ is a call to fib(k). Leaf nodes ($k \leq 1$) are base cases. Total: 15 nodes.

fib(5) ├── fib(4) │ ├── fib(3) │ │ ├── fib(2) │ │ │ ├── fib(1) → 1 │ │ │ └── fib(0) → 0 │ │ └── fib(1) → 1 │ └── fib(2) │ ├── fib(1) → 1 │ └── fib(0) → 0 └── fib(3) ├── fib(2) │ ├── fib(1) → 1 │ └── fib(0) → 0 └── fib(1) → 1

fib(3) appears twice, fib(2) three times. The same sub-problems are solved repeatedly — a hallmark of exponential inefficiency. This redundancy is what the lower bound below captures.

Part 3 — Proof that node count $T(n) \geq \varphi^{n-1}$

Let $T(n)$ = total nodes in the call tree of fib(n).

Claim: $\forall\, n \geq 1:\; T(n) \geq \varphi^{n-1}$. p. 31 — strong induction template

Proof — strong induction on $n$
Base Cases — $n = 1$ and $n = 2$

$n = 1$: fib(1) is a base case, no sub-calls. $T(1) = 1 \geq \varphi^0 = 1$. ✓

$n = 2$: fib(2) calls fib(1) and fib(0). $T(2) = 3 \geq \varphi^1 \approx 1.618$. ✓

Inductive Hypothesis

Suppose $n \geq 3$ is arbitrary but fixed. Assume $T(k) \geq \varphi^{k-1}$ for all $1 \leq k < n$.

Inductive Step

The call tree for fib(n) has one root, plus the full sub-trees of fib(n-1) and fib(n-2):

$$T(n) = 1 + T(n-1) + T(n-2)$$

Since $1 \geq 0$, drop the constant $1$ to get a lower bound, then apply the inductive hypothesis to both sub-calls ($n-1 < n$ and $n-2 < n$):

$$T(n) \;\geq\; T(n-1)+T(n-2) \;\geq\; \varphi^{n-2}+\varphi^{n-3} \;=\; \varphi^{n-3}(\varphi+1)$$

Apply the given identity $\varphi^2 = \varphi + 1$ (the hint):

$$T(n) \;\geq\; \varphi^{n-3}\cdot\varphi^2 \;=\; \varphi^{n-1} \qquad \checkmark$$
□
Part 4 — Implication for running time

From the lecture (slide p. 38): each C++ statement takes a constant amount of time, independent of hardware or compiler. Every node performs $O(1)$ work (one comparison, one addition). So the running time $\propto T(n) \geq \varphi^{n-1}$.

$$\text{Running time of } \texttt{fib}(n) \;=\; \Omega(\varphi^n)$$

This is exponential in $n$. The lecture's table (Problem 1) shows $2^n$ reaches only $n = 19$ per second; $\varphi^n \approx 1.618^n$ is similarly catastrophic. An iterative Fibonacci runs in $\Theta(n)$, processing $10^6$ values per second.

Per slide p. 6: the input entirely determines the running time. Here the input is just the integer $n$, so there is no best/worst case distinction (slide pp. 44–47) — the running time is exactly a function of $n$ alone.

Analysis of is_sorted()

Statement counting · Exact running time · Best/worst case  ·  Lecture slides pp. 37–47, 56–62

The lecture's method. Slides pp. 39–41 analyse insertion_sort() by assigning a cost constant $c_i$ to each statement and counting how many times it executes. The total running time is $\sum c_i \cdot (\text{count}_i)$. This technique is used here verbatim.

Best/worst case — precise definitions (slide p. 46): "Best case: minimum running time over all inputs where the parameter value is $n$. Worst case: maximum running time over all inputs where the parameter value is $n$." The parameter here is the array length $n$.

Finding the dominant term. Slide p. 61 advises: "Focus on code hotspots — if code-units $A$ and $B$ take constant time, and $A$ always runs more times than $B$, then ignore $B$." And slide p. 56: "Focus on the fastest growing term of a function ignoring the constant coefficient." These are the justifications for dropping lower-order terms in the final $\Theta$ expression.

Warning on imprecision. Slide p. 57: "Saying 'an algorithm has a running time of $O(n^2)$' is not a mathematically precise statement." We therefore always say: the best-case (or worst-case) running time of is_sorted() is $\Theta(\cdot)$.

Lecture refs: insertion_sort analysis method (pp. 39–41); best/worst-case definitions (pp. 44–47); dominant term / hotspot tips (pp. 61–62); $\Theta$ definition (p. 50); imprecision warning (p. 57). 
bool is_sorted(int arr[], int n) {
    for(int i = 0; i < n - 1; ++i) {
        if(arr[i] > arr[i + 1]) {
            return false;
        }
    }
    return true;
}
Loop count in terms of $a$

Let $a$ be the value of i when the function returns.

  • Returns false on iteration $i = a$: Loop ran for $i = 0, 1, \ldots, a$, so $a + 1$ times.
  • Returns true: Loop ran for all $i = 0, \ldots, n-2$, so $n-1$ times. (Here $a = n-1$ is the first value where the loop condition $i < n-1$ fails.)
Exact running time — following the lecture's per-statement method pp. 39–41

The condition is checked once more than the increment (the last check fails the loop, so no increment follows) — the same pattern as the outer for-loop of insertion_sort in the lecture (slide p. 39: i < n is checked $n$ times, ++i fires $n-1$ times).

Best Case — array already sorted (or $n = 1$)

No element is out of order; the loop completes all iterations without returning false. The condition $c_2$ fires $n$ times (last check fails), increment $c_3$ fires $n-1$ times, comparison $c_4$ fires $n-1$ times.

$$T_\text{best}(n) = c_1 + n\,c_2 + (n-1)c_3 + (n-1)c_4 + c_6 = (c_2+c_3+c_4)\,n + \text{const}$$

Dominant term is linear in $n$. Best case $\Theta(n)$

Worst Case — first pair out of order ($a = 0$, i.e., arr[0] > arr[1])

The condition fires once ($c_2$), the comparison fires once and is true ($c_4$), the function returns false ($c_5$). No increment fires.

$$T_\text{worst}(n) = c_1 + c_2 + c_4 + c_5 = \text{const (independent of } n)$$

Worst case $\Theta(1)$

Slide p. 57 warns against imprecision: $\Theta$ alone says nothing about best or worst case. Here the best case is $\Theta(n)$ and the worst case is $\Theta(1)$ — the opposite of what one might naively guess. An unsorted array (worst input for the sorting task) actually gives the fastest exit from this check.

Analysis of func()

Input-dependent branching · Best/worst inputs · Upper bounds  ·  Lecture slides pp. 8–9, 44–47, 50–52, 60–62

Input-dependent cost. Slide p. 8 introduces compute_sum() where the inner loop runs arr[i] times — the cost depends on array values, not just length. The lecture asks (slide p. 9): "How much slower is compute_sum() than compute_sum_2()?" — its best case is $\Theta(n)$ but worst case is unbounded (slide p. 63). Similarly here, which branch fires depends on values, but at least the total number of iterations is fixed.

Precise best/worst case. Slide p. 62: "Determine the worst (or best) case input — you need to be clear about the parameter for which you are determining the worst (or best) case!" Here the parameter is $n$ (array length). We ask: over all arrays of length $n$, which minimises / maximises the running time?

Tip from slide p. 62: To find the worst-case input, determine which code path (hotspot) executes the maximum number of times. Then ask: what input causes that path to fire every iteration?

Checking upper bounds. The Big-$O$ definition (slide p. 50): $f(n) = O(g(n))$ iff $\exists\, c > 0, n_0 \geq 0$ such that $f(n) \leq c\,g(n)$ for all $n \geq n_0$. We check this definition directly with the specific proposed bound $c_1 n / 10$.

Lecture refs: compute_sum input-dependent analysis (pp. 8–9, 63); best/worst-case definitions (pp. 44–47); Big-O definition (p. 50); hotspot / dominant term tips (pp. 61–62). 
void func(int arr[], int n) {
    for(int i = 0; i < n - 2; ++i) {
        if(arr[i] > arr[i+1] && arr[i+1] < arr[i+2]) {
            do_whats_needed_in_first_if();     // cost c₁ (very expensive)
        } else if (arr[i] > arr[i+1]) {
            do_whats_needed_in_second_if();    // cost c₂ (moderately expensive)
        } else {
            do_whats_needed_in_else();         // cost c₃ (very cheap)
        }
    }
}

Let $a_1, a_2, a_3$ = number of times first-if, second-if, else fire. Since every iteration runs exactly one branch: $a_1 + a_2 + a_3 = n-2$. Given: $100c_3 < c_2 < c_1/100$, so $c_3 < c_1/10{,}000$.

Simple upper and lower bounds
$$(n-2)\,c_3 \;\leq\; T(n) \;\leq\; (n-2)\,c_1$$

Both are $\Theta(n)$ — but the ratio of upper to lower bound exceeds $c_1/c_3 > 10{,}000$.

Best and worst case inputs and running times
Best Case input — non-decreasing array

The cheapest branch is else ($c_3$). It fires when arr[i] <= arr[i+1], which holds for all $i$ in a non-decreasing array. Then $a_1 = a_2 = 0$, $a_3 = n-2$.

$$T_\text{best}(n) = (n-2)\,c_3 \quad\Rightarrow\quad \text{best-case running time } \Theta(n)$$
Worst Case input — alternating array

The most expensive branch is first-if ($c_1$). Its condition requires arr[i] > arr[i+1] AND arr[i+1] < arr[i+2] — a "valley" at position $i+1$. An alternating array $[L, S, L, S, \ldots]$ (with $L \gg S$, e.g. $[100, 1, 100, 1, \ldots]$) satisfies this at every $i$. Then $a_1 = n-2$.

$$T_\text{worst}(n) = (n-2)\,c_1 \quad\Rightarrow\quad \text{worst-case running time } \Theta(n)$$
Is $c_1 n/10$ an upper bound? — applying the Big-$O$ definition p. 50

For the best case: $T_\text{best}(n) = (n-2)c_3 < n\,c_3$. Since $c_3 < c_1/10{,}000 < c_1/10$, we have $T_\text{best}(n) < n\cdot c_1/10$. So yes, $c_1 n/10$ is an upper bound for the best-case running time. Yes — upper bound for best case

For the worst case: $T_\text{worst}(n) = (n-2)c_1$. For $n \geq 3$: $(n-2)c_1 \geq c_1 n/2 > c_1 n/10$. So $c_1 n/10$ is not an upper bound for the worst-case running time. No — not an upper bound for worst case

Selection Sort

Nested loops · $\Theta(n^2)$ in all cases · Correctness by induction  ·  Lecture slides pp. 29–36, 39–47, 56–62

Nested loop analysis. The lecture's "One Loop Guide" (slide p. 60): if a loop runs $n$ times with constant-time body, total cost is $\Theta(n)$. For nested loops, we sum the inner count over all outer iterations — exactly how the lecture derives the $\sum_{i=1}^{n-1} t_i$ term for insertion sort (slides pp. 39–41). Here we get $\sum_{i=0}^{n-2}(n-1-i) = n(n-1)/2$.

$O(n^2)$: From slide p. 50, $f = O(g)$ means $f$ is at most a constant multiple of $g$ for large $n$. If no statement runs more than $n^2$ times, the algorithm is $O(n^2)$.

$\Omega(n^2)$: From slide p. 50, $f = \Omega(g)$ means $f$ is at least a constant multiple of $g$. Finding one specific statement that provably runs $\geq cn^2$ times — regardless of input — proves $\Omega(n^2)$.

$\Theta(n^2)$ in all cases. Selection sort's inner comparison always runs exactly $n(n-1)/2$ times, regardless of the input's values or order. This is unlike insertion sort, where the inner loop count depends on the number of inversions $t_i$ (slide p. 39).

Correctness model. The lecture proves insertion sort correct via a predicate about the sorted prefix (slides pp. 29–36, 59). We use the same pattern: state $P(i)$, prove base case and inductive step. The lecture frames this as finite-domain induction (slide p. 32): $i$ ranges over $\{0, \ldots, n-1\}$.

Lecture refs: one-loop guide / nested loops (pp. 60–62); $O/\Omega/\Theta$ definitions (pp. 50–52); dominant term theorem (p. 56); insertion_sort correctness as model (pp. 29–36, 59); finite-domain induction (p. 32). 
void selection_sort(int arr[], int n) {
    for(int i = 0; i < n - 1; ++i) {          // outer: runs n-1 times
        int candidate = i;
        for(int j = i + 1; j < n; ++j) {         // inner: runs n-1-i times
            if(arr[j] < arr[candidate]) candidate = j;
        }
        std::swap(arr[i], arr[candidate]);
    }
}
Loop counts

Outer loop: $i = 0, 1, \ldots, n-2$ — exactly $n-1$ times.

Can we ignore int candidate = i? Yes. Slide p. 61: "If code-units $A$ and $B$ take constant time, and $A$ always runs more times than $B$, then ignore $B$." The statement int candidate = i runs $n-1$ times (cost $O(n)$), dominated by the inner comparison which runs $\frac{n(n-1)}{2}$ times (cost $\Theta(n^2)$). So int candidate = i is not the hotspot.

Inner loop: For outer iteration $i$, inner loop runs $n-1-i$ times. Total comparisons:

$$\sum_{i=0}^{n-2}(n-1-i) = (n-1)+(n-2)+\cdots+1 = \frac{n(n-1)}{2}$$

This is the same arithmetic sum as in the insertion sort analysis on slide p. 41 (where the lecture sums $t_i$ over $i$).

Is it $O(n^2)$?

Each statement in the outer loop body executes at most $n-1 \leq n$ times per outer iteration, and there are $n-1 \leq n$ outer iterations. No statement executes more than $n^2$ times. $O(n^2)$ — slide p. 50

Is it $\Omega(n^2)$?

The comparison arr[j] < arr[candidate] runs exactly $\frac{n(n-1)}{2}$ times for every input of size $n$ — no early exit is possible. For $n \geq 2$: $\frac{n(n-1)}{2} \geq \frac{n^2}{4}$. The running time is at least $\frac{c_\text{cmp}}{4}\,n^2$. $\Omega(n^2)$ — slide p. 50

Combining: Best case $\Theta(n^2)$ Worst case $\Theta(n^2)$ — selection sort is $\Theta(n^2)$ on every input.

Correctness — mirroring the lecture's insertion sort proof pp. 29–36, 59
Predicate $P(i)$ — loop invariant

After $i$ iterations of the outer for-loop: arr[0..i-1] contains the $i$ smallest elements of the original array in sorted (non-decreasing) order, each in its final position.

Claim: $P(i)$ holds for all $i = 0, \ldots, n-1$ (finite domain, slide p. 32). At $i = n-1$ this gives a fully sorted array.

Proof — induction on $i$ over $\{0,\ldots,n-1\}$ p. 32 — finite domain template
Base Case — $P(0)$, before any iterations

arr[0..-1] is empty. $P(0)$ holds vacuously. ✓

Inductive Hypothesis — assume $P(i)$

Suppose $i$ is arbitrary but fixed, $0 \leq i \leq n-2$. Assume $P(i)$: arr[0..i-1] holds the $i$ smallest elements in sorted order, finalised.

Inductive Step — prove $P(i+1)$ after the body of iteration $i$

The inner loop scans arr[i..n-1]. Initially candidate = i; whenever a smaller element is found at position $j$, candidate is updated. After the inner loop, arr[candidate] $\leq$ arr[j] for all $j \in \{i, \ldots, n-1\}$ — i.e., arr[candidate] is the minimum of arr[i..n-1].

The swap places this minimum at position $i$. Combined with $P(i)$, arr[0..i] now contains the $i+1$ smallest elements in sorted order, each finalised. $P(i+1)$ holds. ✓

Termination — $P(n-1)$ after $n-1$ outer iterations

arr[0..n-2] contains the $n-1$ smallest in sorted order. The remaining arr[n-1] is the largest and is already correctly placed. The array is fully sorted. ✓

□

Selection Sort with Early Checking

Sub-routine cost in a loop · Best/worst case  ·  Lecture slides pp. 44–47, 60–62

Sub-routine cost inside a loop. The lecture's "One Loop Guide" (slide p. 60): if the loop body takes constant time $c$ per iteration and runs $n$ times, total is $\Theta(cn) = \Theta(n)$. When the body calls a sub-routine that takes $\Theta(n)$ itself, the loop now does $\Theta(n)$ per iteration — total $\Theta(n^2)$. We must account for which case of is_sorted applies, since it can be $\Theta(1)$ or $\Theta(n)$.

Key question: when is_sorted is called inside the loop, how quickly does it return? This depends on the state of the array at the time of the call — which changes as sorting progresses. We reason about this state for each scenario.

Lecture refs: one-loop guide (p. 60); sub-routine analysis (pp. 61–62); best/worst-case definitions (pp. 44–47); dominant term (p. 56). 
void selection_sort(int arr[], int n) {
    for(int i = 0; i < n - 1; ++i) {
        if(is_sorted(arr, n)) return;   // Θ(n) or Θ(1) depending on array state
        int candidate = i;
        for(int j = i + 1; j < n; ++j) {
            if(arr[j] < arr[candidate]) candidate = j;
        }
        std::swap(arr[i], arr[candidate]);
    }
}
Best Case

Input: already sorted array. On $i = 0$, is_sorted(arr, n) scans all $n-1$ adjacent pairs (all are in order) and returns true — this is the best case of is_sorted from Problem 3, which costs $\Theta(n)$. The outer function returns immediately.

$$T_\text{best}(n) = \Theta(n)$$

This is a dramatic improvement over the original selection sort's best case of $\Theta(n^2)$.

Worst Case

Input: reverse-sorted array $[n, n-1, \ldots, 2, 1]$. is_sorted is called at the start of each outer iteration. At outer iteration $i$, the loop invariant from Problem 5 tells us arr[0..i-1] is sorted, but arr[i..n-1] is still in some order derived from the reverse-sorted original. In particular, after $i$ swaps on a reverse-sorted array, arr[i] > arr[i+1] is still true (the remaining suffix is still partially reversed). So is_sorted detects the first out-of-order pair immediately — this is the worst case of is_sorted, costing $\Theta(1)$.

  • $n-1$ calls to is_sorted, each $\Theta(1)$: total $\Theta(n)$.
  • Inner loop still runs $\frac{n(n-1)}{2}$ times: total $\Theta(n^2)$.
$$T_\text{worst}(n) = \Theta(n^2) + \Theta(n) = \Theta(n^2)$$

The $\Theta(n^2)$ inner loop dominates — per slide p. 56 (focus on the dominant term). Best case $\Theta(n)$ Worst case $\Theta(n^2)$

Bubble Sort

Shrinking inner loop · $\Theta(n^2)$ always · Correctness by induction  ·  Lecture slides pp. 29–36, 39–47, 56–62

Same analysis pattern as selection sort. We sum inner iterations over outer, identify a statement running $\Theta(n^2)$ times unconditionally, and use the formal $\Theta$ definition (slide p. 50). The correctness proof follows the same finite-domain induction structure as the lecture's insertion sort proof (slides pp. 29–36, 32).

Why the loop shrinks — a loop invariant. Slide p. 34: "Typical claim: after step $i$, some property holds." Here the property is: "After outer pass $i$, the $i+1$ largest elements are in their final positions at the end." This invariant is what justifies the shrinking bound $n-i-1$ — there is no need to re-examine elements already in their final positions.

No best-case improvement here. Unlike insertion sort (whose inner loop stops early when no shifts are needed) or the version from Problem 6 (which checks is_sorted), this bubble sort has no early-exit mechanism. The inner loop always runs to completion.

Lecture refs: nested loop analysis (pp. 60–62); correctness by induction model (pp. 29–36, 59); finite domain induction (p. 32); $O/\Omega/\Theta$ definitions (pp. 50–52). 
void bubble_sort(int arr[], int n) {
    for (int i = 0; i < n - 1; ++i) {
        for (int j = 0; j < n - i - 1; ++j) {  // bound shrinks: n-1, n-2, …, 1
            if (arr[j] > arr[j + 1]) {
                std::swap(arr[j], arr[j + 1]);
            }
        }
    }
}
Why does the inner loop run $n - i - 1$ times?

The loop invariant (proved below) guarantees: after outer pass $i$, the $i+1$ largest elements occupy positions $n-i-1, \ldots, n-1$ in sorted order, finalised. Comparing against finalised positions is unnecessary — they cannot change further. So pass $i+1$ only needs to examine $j = 0, \ldots, n-i-2$, giving exactly $n-i-1$ comparisons. The bound $j < n-i-1$ enforces this.

Running time

The comparison inside the inner loop fires once per inner iteration, regardless of whether a swap occurs. Total comparisons (same sum as selection sort):

$$\sum_{i=0}^{n-2}(n-1-i) = (n-1)+(n-2)+\cdots+1 = \frac{n(n-1)}{2} = \Theta(n^2)$$

This count is fixed for every input of size $n$. By the same $O(n^2)$ and $\Omega(n^2)$ argument as Problem 5:

$$T_\text{best}(n) = T_\text{worst}(n) = \Theta(n^2)$$

An optimised version adds a boolean "swapped" flag: if no swaps occur during a pass, the array is already sorted, and the algorithm exits in $\Theta(n)$ best case. This version as written lacks that optimisation.

Correctness — following the lecture's induction model pp. 29–36, 59
Predicate $P(i)$ — loop invariant

After $i$ iterations of the outer for-loop: arr[n-i..n-1] contains the $i$ largest elements of the original array in sorted (non-decreasing) order, each in its final position.

Proof — induction on $i$ over $\{0,\ldots,n-1\}$ p. 32 — finite domain induction
Base Case — $P(0)$, before any iterations

arr[n..n-1] is empty. $P(0)$ holds vacuously. ✓

Inductive Hypothesis — assume $P(i)$

Suppose $i$ is arbitrary but fixed, $0 \leq i \leq n-2$. Assume $P(i)$: arr[n-i..n-1] holds the $i$ largest elements, finalised.

Inductive Step — prove $P(i+1)$ after outer iteration $i$

The inner loop scans $j = 0, \ldots, n-i-2$, comparing and potentially swapping adjacent pairs in arr[0..n-i-1].

Claim: after the inner loop, the maximum of arr[0..n-i-1] is at position $n-i-1$.

Sub-argument: Let $M$ be the maximum value in arr[0..n-i-1]. At any step $j$ where $M$ sits at position $j$: since $M \geq$ all elements, arr[j] > arr[j+1], so a swap occurs and $M$ moves to $j+1$. This continues until $j = n-i-2$, after which $M$ lands at $n-i-1$. (Other swaps may occur, but $M$ always wins each comparison it participates in.)

Therefore arr[n-i-1] holds the maximum of arr[0..n-i-1]. Combined with $P(i)$, arr[n-i-1..n-1] now holds the $i+1$ largest in non-decreasing order. $P(i+1)$ holds. ✓

Termination — $P(n-1)$ after $n-1$ outer iterations

arr[1..n-1] holds the $n-1$ largest in sorted order. arr[0] is the minimum and is correctly placed. The array is fully sorted. ✓

Proving $f(n) = \Theta(g(n))$

Formal $\Theta$ definition · Little-$o$ limit form · Lecture slides pp. 50–52, 56

Formal $\Theta$ definition (slide p. 50): $f(n) = \Theta(g(n))$ iff $\exists\, c_1 > 0,\, c_2 > 0,\, n_0 \geq 0$ such that $c_1\,g(n) \leq f(n) \leq c_2\,g(n)$ for all $n \geq n_0$. See also slide p. 51 (the three-region diagram illustrating this sandwich).

Little-$o$ and the limit (slide p. 52): $h(n) = o(g(n))$ iff $\lim_{n\to\infty} h(n)/g(n) = 0$. This is precisely the condition given in the problem: $h$ grows strictly slower than $g$ and becomes negligible relative to $g$.

The recipe from the lecture (slide p. 56, dominant-term theorem proof): to show $an^k + \text{lower terms} = \Theta(n^k)$, the lecture constructs explicit constants $c_1, c_2, n_0$ and verifies both the upper and lower inequalities. The homework hint says: "Just follow the recipe in the proof from Lecture 2." That recipe is: use the limit condition to bound $h$ in terms of $g$ by a fraction $\varepsilon$ of $g$, then substitute into $f = cg + h$ to obtain both inequalities.

Connection to slide p. 56. In the dominant-term theorem, $f(n) = a_k n^k + \cdots + a_0$. Setting $c = a_k$, $g(n) = n^k$, and $h(n) = a_{k-1}n^{k-1}+\cdots+a_0$, we have $h(n)/g(n) \to 0$ as $n\to\infty$. Our result is the general version of that specific fact.

Lecture refs: Big-$O$, $\Omega$, $\Theta$ definitions (p. 50); $\Theta$ diagram (p. 51); little-$o$ limit form (p. 52); dominant-term proof recipe (p. 56); intuitive notation table (p. 53).
Given and To Prove

Given: $f(n) = c\,g(n) + h(n)$ where $c > 0$, $g(n) \geq 0$, and $\displaystyle\lim_{n\to\infty}\frac{h(n)}{g(n)} = 0$.

To prove: $f(n) = \Theta(g(n))$ — i.e., find $c_1, c_2 > 0$ and $n_0$ such that $c_1\,g(n) \leq f(n) \leq c_2\,g(n)$ for all $n \geq n_0$. p. 50 — $\Theta$ definition

Proof — following the recipe from slide p. 56

  1. Translate the limit condition into a concrete bound. The condition $\lim_{n\to\infty} h(n)/g(n) = 0$ means $h = o(g)$ (slide p. 52 — little-$o$ via limit). By the definition of the limit: for any $\varepsilon > 0$, $\exists\, N$ such that $\forall\, n \geq N$: $\left|\frac{h(n)}{g(n)}\right| < \varepsilon$. Since $g(n) \geq 0$, multiplying gives:

    $$-\varepsilon\,g(n) \;<\; h(n) \;<\; \varepsilon\,g(n) \qquad \text{for all } n \geq N$$

  2. Choose $\varepsilon = c/2$. This is the key step of the recipe — choosing $\varepsilon$ so that the additive perturbation $h$ is bounded by half the leading coefficient. There exists $N$ (set $n_0 = N$) such that for all $n \geq n_0$:

    $$-\frac{c}{2}\,g(n) \;<\; h(n) \;<\; \frac{c}{2}\,g(n)$$

  3. Prove the upper bound $f(n) \leq c_2\,g(n)$. Using the right inequality $h(n) < \frac{c}{2}g(n)$:

    $$f(n) = c\,g(n)+h(n) \;<\; c\,g(n)+\frac{c}{2}g(n) = \frac{3c}{2}\,g(n)$$

    Set $c_2 = 3c/2 > 0$. Then $f(n) \leq c_2\,g(n)$ for all $n \geq n_0$. ✓

  4. Prove the lower bound $f(n) \geq c_1\,g(n)$. Using the left inequality $h(n) > -\frac{c}{2}g(n)$:

    $$f(n) = c\,g(n)+h(n) \;>\; c\,g(n)-\frac{c}{2}g(n) = \frac{c}{2}\,g(n)$$

    Set $c_1 = c/2 > 0$. Then $f(n) \geq c_1\,g(n)$ for all $n \geq n_0$. ✓

  5. Conclude by the $\Theta$ definition (slide p. 50). We have exhibited $c_1 = c/2 > 0$, $c_2 = 3c/2 > 0$, and $n_0 = N$ such that:

    $$\frac{c}{2}\,g(n) \;\leq\; f(n) \;\leq\; \frac{3c}{2}\,g(n) \qquad \forall\, n \geq n_0$$

    Therefore $f(n) = \Theta(g(n))$. $\qquad$ □

□

Why $\varepsilon = c/2$ specifically? Any $\varepsilon < c$ would work. The choice $c/2$ is symmetric: it makes the lower bound coefficient $c - c/2 = c/2$ and the upper bound $c + c/2 = 3c/2$, both positive, and gives a clean ratio $c_2/c_1 = 3$. Choosing $\varepsilon = c/10$ would give $c_1 = 9c/10$, $c_2 = 11c/10$ — tighter but equivalent.

Intuitive meaning (slide p. 53): $\Theta$ means both functions "grow at the same rate." Here $h = o(g)$ means $h$ grows strictly slower — it is eventually negligible. Adding a negligible term $h$ to $c\,g$ cannot change the growth rate of the result: it only perturbs the leading constant, which $\Theta$ absorbs.
